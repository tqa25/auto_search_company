**작성일:** 2026년 4월 13일  
**버전:** 3.0 — Firecrawl + Google Gemini AI 하이브리드 자동화

## 1. 프로젝트 요약 (경영진 보고용)

### 목표
인터넷에서 **6,000개 이상의 기업**의 연락처 정보(주소, 전화번호, 이메일, 웹사이트, 대표자)를 자동으로 검색 및 수집하는 툴을 구축하여, 기존의 수동 검색 프로세스를 대체합니다.

### 전체 프로세스

* **1단계: 명단 읽기** - 시스템이 6,000개 이상의 기업 이름이 포함된 Excel 파일을 열고 순차적으로 읽습니다.
* **2단계: 자동 검색** - 시스템이 각 기업 이름을 구글(Google)에 자동 검색하고 상위 10개의 결과 링크를 가져옵니다. **(이때 확보한 10개의 링크 데이터는 즉시 데이터베이스에 저장됩니다.)**
* **3단계: 내용 수집 (Scrape)** - 10개의 링크 중 9개의 주요 타겟 타겟 웹사이트이거나 해당 기업의 공식 웹사이트로 연결되는 링크만 선별하여 해당 페이지 내부의 텍스트(광고 제거됨)를 수집합니다. **(페이지별 텍스트 데이터는 즉시 데이터베이스에 보존됩니다.)**
* **4단계: AI 분석 및 분류** - 인공지능(Google Gemini)이 3단계에서 수집된 텍스트를 분석하여 주소, 전화번호, 이메일, 팩스, 웹사이트, 대표자 정보를 추출합니다. **(각 정보가 어느 출처에서 왔는지 명확히 기록됩니다.)**
* **5단계: 결과 출력** - 모든 결과가 Excel 파일로 기록됩니다. 각 기업마다 **모든 출처에서 수집된 전체 정보가 표시되며**, 사용자가 쉽게 파악할 수 있도록 분류됩니다.

> **💡 경영진 참고사항: 왜 10개 → 100개 → 1,000개 → 6,000개로 나누어서 점진적으로 테스트합니까?**
> 업무 자동화 프로젝트에서 "점진적 인상(Incremental Rollout)"은 필수 방어 기제입니다.
> 1. **비용 한도 방어:** 로직 오류 상태로 6,000개를 일시에 쏘아 올릴 경우, 불과 몇 시간 만에 크레딧(자금)이 모조리 허공으로 증발해 버립니다.
> 2. **예외/이상 엣지 케이스 추적:** 동명이인(다른 기업), 페이스북 주소만 존재하는 유령 기업 등, 적은 샘플에서는 찾지 못한 돌발 상황들을 단계별로 식별해 AI를 똑똑하게 가르쳐야 합니다.
> 3. **서버 방어 및 블랙리스트 회피:** 무턱대고 빠른 속도로 돌리면 타겟 사이트(예: masothue)나 구글 보안 프로그램이 디도스(DDoS) 공격으로 간주하고 자사 아이피(IP)를 완전히 영구 차단시켜버릴 위험이 있으므로, 안정적인 속도를 1,000개 단위에서 찾아내야 합니다.

## 4. 로그(Log) 백그라운드 기록 시스템

로그 시스템은 다음과 같은 엄격한 2가지 철칙이 적용됩니다.
1. **AI 기계 가독성 (Machine Readable)** — 향후 AI가 직접 로그 스프레드시트를 수거하여 통계와 경향성을 뽑을 수 있도록 규격화/표준화합니다.
2. **원시 데이터 (Raw Data) 절대 보존** — 검색 기록, 가져온 스크래핑 문서 전문 등 과정 단계 데이터는 버리지 않고 로컬 DB에 보존합니다. 이후 재분석이 필요할 시 Firecrawl 과금 없이 무상으로 AI만 사용하여 재작업이 가능하도록 합니다.

### 4.1. 기업별 독립 상세 로그 예시
| 열(Column) 헤더 명칭 | 저장 대상 | 데이터 예시 |
|---|---|---|
| company_id | 자사 인덱스 고유 번호 | CMP-0001 |
| company_name | 명부에서 읽어들인 이름 | ABC Limited |
| step | 트리거 중인 단계 | search / filter / scrape / ai_extract |
| started_at | 시작 Timestamp | 2026-04-20 09:15:23 |
| duration_seconds | 실행 경과 시간 | 22 |
| source_url | 처리 대상 타겟 링크 | https://masothue.com/abc... |
| credits_used | 해당 Step에서 소모한 비용 | 3 |
| status | 처리 상태 확인 | success / skipped / failed |

### 4.2. 일일 경영진 보고용 종합 로그 지표
* **일일 진척률 (Progress):** 당일 처리 건수 및 전체 대비 총 처리 진척률 (예: 총 2,340 / 6,000 = 39%)
* **데이터 확보율 (Hit Ratio):** 유효한 연락처 정보가 하나 이상 잡힌 기업의 비율 (예: 78%)
* **예산 체력도 (Credit Burn Rate):** 기 사용된 크레딧 대조 잔여 크레딧
* **최다 병목 에러 현황:** 프로젝트 진행 중 가장 많이 나타나는 에러 상위 5건 분석

| Search (검색) | 기업별 최상단 결과 링크 10개 + Title + 스니펫 | Firecrawl 크레딧을 추가 소모하지 않고, 나중에 완전히 다른 종류의 웹사이트를 분석 타겟에 추가할 때 재활용 목적. |
| Scrape (콘텐츠 수집) | 확보된 웹페이지 텍스트(Markdown 포맷 완성본) 전문 | **프로젝트에서 가장 비용이 많이 투자된 핵심 자원.** 나중에 AI 모델을 Gemini에서 다른 모델로 바꾸거나, 수집 조항(예: 창립 연도 추가 조사)을 바꿀 때, 재스크래핑 요금 없이 무제한 재분석 가능. |
| Extract (AI 추출) | 파싱(Parsing)이 완료된 JSON 구조화 데이터 | 향후 프롬프트 엔지니어링 성능 비교 및 백테스트용 (히스토리성 자료). |

## 7. 주요 리스크 관리 및 대응책 (Risk Management)

| 위험 요소 (Risk) | 등급 | 사전 대응 방안 |
|---|---|---|
| 인터넷 디지털 발자국(Digital Footprint)이 전혀 없는 소형 로컬 기업 비율 발생 (약 20~30% 추정) | 높음 (High) | 시스템에서 신속히 "검색 불가" 라벨을 부착하여 빠르게 Pass 시킨 후, 최종 단계에서 실패 명단만 시트로 빼 인력 리서치 팀에게 이관. |
| 페이스북/링크드인 등 SNS 플랫폼의 봇(Bot) 감쇄 및 차단 방어막 | 보통 (Mid) | 해당 2곳은 Core 타겟이 아닌 보조적 목적임. 타임아웃 발생 시 즉시 무시(Skip) 후 넘어가도록 에러 로직 처리. |
| 대규모 트래픽에 의한 타겟 웹사이트 및 구글 자동 차단 (HTTP 429 에러 발생) | 보통 (Mid) | 1,000건 테스트 시 랜덤 Sleep 딜레이 시간 적용, 요청량 제어 (Rate Limiting) 매커니즘 코드에 탑재. |
| 가동 중 윈도우 프리징, 정전, 네트워크 에러 등 발생 시 | 낮음 (Low) | **세션(Session) Resume 프로토콜 무조건 적용.** 스크립트 중단 시, 제일 마지막 처리된 기업의 ID를 기억해 익일 바로 이어서 출발. |
| AI 환각 (오인식 - 이메일과 팩스 혼동 등) 현상 | 낮음 (Low) | 100건 테스트 구간에서 엄격한 AI 프롬프트 체인 튜닝. Excel 출력 시 출처를 의무로 기재하게 하여, 인간 작업자가 빠르게 눈으로 훑으며 검증(Human in the loop) 가능토록 조치. |
